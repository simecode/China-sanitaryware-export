# 修复说明（用你真实的代码 + 数据实测过）

替换仓库里的两个文件即可：**`app.py`** 和 **`prepare_default_data.py`**，
并把 **`requirements.txt`** 也换上（补了 `pyarrow`，否则 Streamlit Cloud 读 parquet 会报错）。

## 修复了三个叠加的 Bug（都已实测验证）

1. **重复计数（在 prepare_default_data.py）**
   原脚本会把 `6910_2026_1-4月.xlsx`(月度,12.78亿) 和 `6910_2026.xlsx`(年度,8.35亿)
   **相加**，2026 变成 21 亿。新脚本给每行打了「数据粒度」标记，
   **同一年若同时有月度和年度文件，自动丢弃年度行、以月度为准**，不再重复。

2. **伪造月份（在 prepare_default_data.py）**
   原脚本对没有月份的文件强行写成 `YYYY-01`，把全年数据伪装成 1 月。
   新脚本只在真有「数据年月」时解析出 1–12 的真实月份，否则月份留空、标为「年度」。

3. **同比口径错误（在 app.py，首页“暴跌”的根因）**
   原 `calc_metrics` 拿 2026 全部数据(只有几个月) 比 2025 全年。
   新版改为**同口径**：找出最新年份覆盖的月份(如 1–4 月)，
   把**带月份的其它年份也截到相同月份**再比；
   **缺月度数据的年份不参与同比**（宁可显示“无数据”，也不给错误暴跌）。

## 实测结果（用你上传的三个文件）
- 2026 前4月总额 = **12.78 亿美元**（不再是 21 亿，也不再被 2025 全年错误对比）。
- 前五大目的地：美国、越南、新加坡、泰国、马来西亚。
- 因目前缺 2025 月度数据，2026 同比显示“无月度数据”——**这是对的**，不是 bug。
- 我临时造了一个“2025 年 1–4 月”样例验证：补上后 **2026 vs 2025 同口径同比自动出现（+11.11%）**。

## 你需要做的一件事（数据，代码替不了）
要让“前 4 月同比”真正出现，**每个想对比的年份都要有对应的月度文件**，
格式与 `6910_2026_1-4月.xlsx` 一样（含 `数据年月` 列，如 202501、202502…），
命名例如 `6910_2025_1-4月.xlsx`、`6910_2024_1-4月.xlsx`，放进 `raw_data/`。
> 这正是你已验证的 `6910月度分析.py` 里 `已标准化_前四月` 文件夹的那批数据。

## 操作步骤
1. 用本 `app.py`、`prepare_default_data.py`、`requirements.txt` 替换仓库同名文件。
2. 把各年的月度 Excel 放进 `raw_data/`（区域映射表 `区域映射表.xlsx` 放根目录）。
3. 本地跑一次 `python prepare_default_data.py` 重新生成 `data/*.parquet`。
4. 把 `data/` 下新的 parquet 一起提交：
   `git add -A && git commit -m "fix: 同口径前N月同比+去重+真实月份" && git push`
5. Streamlit Cloud 自动重新部署（或后台 Reboot app）。

## 侧边栏“分析维度”说明
- **同口径(前N月)对比**：默认，首页/各维度都按各年相同月份算同比。
- **月度动态演变**：各年相同月份的月度走势叠加对比。
- **历年全年(完整年份)**：只用年度文件展示完整年份，自动排除不完整的最新年份。
